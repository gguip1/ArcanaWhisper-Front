# ArcanaWhisper 프로젝트 가이드라인

## 문서 목적

이 문서는 ArcanaWhisper 프론트엔드에서 지속적으로 지켜야 하는 공통 규칙의 단일 기준 문서입니다.

- 디자인 스타일
- UI/UX 원칙
- 코드 변경 원칙
- 로컬 훅과 CI 규칙
- 커밋 메시지 규칙

프로젝트의 일시적인 상태, 임시 TODO, 즉시 해결 필요 목록 같은 휘발성 정보는 이 문서에 두지 않습니다.

## 프로젝트 개요

ArcanaWhisper는 AI 기반 타로 리딩 서비스입니다. 사용자가 질문을 입력하고 타로 카드를 선택하면 AI가 해석 결과를 제공합니다.

### 핵심 가치

- **프리미엄**: 고급스럽고 세련된 사용자 경험
- **차분함**: 과하지 않은 절제된 시각적 표현
- **일관성**: 단일 디자인 시스템, 단일 아이콘 세트
- **모바일 우선**: 모든 UI 결정에서 모바일 환경 최우선 고려

## 비협상 규칙

### UI/UX 규칙

1. **이모지 금지**
UI 문자열, 문서 예시, 코멘트, 제품 카피에는 이모지를 사용하지 않습니다.

2. **단일 아이콘 세트**
`react-icons`의 FontAwesome(`fa`, `fa6`)만 사용합니다.

3. **한국어 직접 작성**
서비스 문자열은 한국어로 직접 작성합니다. 별도 i18n 레이어를 다시 도입하지 않습니다.

4. **토큰 우선**
색상, 간격, 타이포그래피는 기존 토큰과 디자인 명세를 먼저 따릅니다.

5. **모바일 우선**
모든 레이아웃과 상호작용은 모바일을 먼저 기준으로 판단합니다.

### 코딩 규칙

1. **최소 변경 원칙**
요청받은 변경만 수행하고, 불필요한 리팩토링은 하지 않습니다.

2. **기존 패턴 준수**
새 패턴을 도입하기 전 기존 코드의 패턴과 구조를 먼저 확인합니다.

3. **과도한 엔지니어링 금지**
현재 필요한 수준을 넘는 추상화나 복잡한 구조를 만들지 않습니다.

4. **관련 문서 동시 업데이트**
작업 방식, 배포 구조, 사용자 경험, 디자인 규칙이 바뀌면 관련 문서도 함께 수정합니다.

## 디자인 시스템

### Celestial 컬러 팔레트

딥 인디고와 코퍼(로즈골드) 조합으로 우주적이고 세련된 분위기를 유지합니다.

| 역할 | 색상 | HEX |
|------|------|-----|
| 배경 | Cosmos | `#0C0C1E` |
| 표면 | Deep Space | `#1A1A2E` |
| 액센트 | Indigo | `#6366F1` |
| 따뜻한 액센트 | Copper | `#B76E79` |
| 텍스트 | Starlight | `#F1F5F9` |

### 시각 스타일 원칙

- 차분하고 절제된 화면을 기본으로 합니다.
- 한 화면에 너무 많은 시각 요소를 쌓지 않습니다.
- 새로운 시각 언어보다 기존 톤을 유지하는 쪽을 우선합니다.
- 반짝임, 별빛, 우주적 분위기는 보조 요소로만 사용합니다.
- 과도한 애니메이션보다 가벼운 전환과 안정적인 가독성을 우선합니다.

### 타이포그래피와 간격

- 모바일에서 읽기 쉬운 크기를 유지합니다.
- 여백은 터치 오류를 줄일 정도로 충분히 둡니다.
- 터치 타겟은 최소 `44x44px`를 기준으로 합니다.
- 기본 레이아웃은 단일 컬럼을 우선하고, 화면이 넓어질 때 점진적으로 확장합니다.

### 아이콘 사용 규칙

허용 예시:

```tsx
import { FaHome, FaArrowRight } from 'react-icons/fa';
```

금지 예시:

```tsx
import { MdHome } from 'react-icons/md'; // 금지
import { IoHome } from 'react-icons/io'; // 금지
```

## 정적 문서와 SEO 처리 방식

### 법적 문서

법적 문서는 React 컴포넌트가 아닌 순수 HTML 파일로 유지합니다.

- `public/privacy-policy.html`
- `public/terms-of-service.html`

이유:

- SEO와 크롤링에 유리함
- 앱 번들과 분리됨
- 법적 문서를 별도로 관리하기 쉬움

### 도메인 처리 원칙

- 앱 코드에서 소비하는 사이트 URL은 환경변수로 관리합니다.
- 정적 SEO 파일은 실제 운영 도메인과 일치해야 합니다.
- 배포 시점에 사용하는 사이트 URL과 정적 파일의 URL이 어긋나지 않게 유지합니다.

## 작업 워크플로우

### 구현 전

- 기존 코드 구조와 패턴을 먼저 확인합니다.
- 애매한 변경이나 구조 변경은 먼저 방향을 정리한 뒤 수정합니다.
- 라이브러리 API나 최신 패턴이 중요하면 공식 문서를 확인합니다.

### PR 전

- `npm run lint`
- `npm run build`

위 두 검증을 통과시키고, 실패 시 규칙을 약화시키기보다 코드를 수정합니다.

## 로컬 가드레일

- 훅 연결: `npm run setup:hooks`
- `pre-commit`: staged 된 `js/jsx/ts/tsx` 파일에 `eslint --fix` 실행 후 다시 stage
- `commit-msg`: 커밋 메시지 형식 강제
- 특별한 이유 없이 `--no-verify`를 사용하지 않습니다.

## 커밋 메시지 규칙

형식:

```text
<emoji> <type>(optional-scope): summary
```

예시:

- `✨ feat: 공유 기능 추가`
- `🐛 fix(auth): 로그인 취소 처리 수정`
- `♻️ refactor: 히스토리 로딩 구조 정리`
- `👷 ci: 프론트 배포 워크플로우 수정`
- `🚀 deploy: S3 배포 단계 조정`

허용 매핑:

- `✨ feat`
- `🐛 fix`
- `♻️ refactor`
- `📝 docs`
- `🎨 style`
- `✅ test`
- `🔧 chore`
- `👷 ci`
- `📦 build`
- `🚀 deploy`
- `⚡ perf`
- `🔥 remove`

예외 허용:

- `Merge ...`
- `Revert ...`
- `fixup! ...`
- `squash! ...`

## CI 규칙

- PR은 `lint`와 `build`를 통과해야 합니다.
- `main`은 같은 검증을 통과한 뒤에만 S3와 CloudFront 배포 단계로 넘어갑니다.
